pelican 内容编写（仅包含markdown方式的内容）

文件元数据

pelican试图从文件系统中获取信息（例如：文章目录），但还是需要你以元数据的形式在文件内提供一些信息。 Markdown 的元数据语法如下：

Title: My super title
Date: 2010-12-03 10:20
Modified: 2010-12-05 19:30
Category: Python
Tags: pelican, publishing
Slug: my-super-post
Authors: Alexis Metaireau, Conan Doyle
Summary: Short version for index and feeds

你也可以有自定义的元数据关键字，只要跟保留的元数据关键字不冲突即可。

Metadata Description
title Title of the article or page
date Publication date (e.g., YYYY-MM-DD HH:SS)
modified Modification date (e.g., YYYY-MM-DD HH:SS)
tags Content tags, separated by commas
keywords Content keywords, separated by commas (HTML content only)
category Content category (one only — not multiple)
slug Identifier used in URLs and translations
author Content author, when there is only one
authors Content authors, when there are multiple。用逗号分隔的列表。
summary Brief description of content for index pages 如果你没有清楚的指定此数据，则将依据设置中的SUMMARY_MAX_LENGTH从文章开始截取数据作为summary
lang Content language ID (en, fr, etc.)
translation Is content is a translation of another (true or false)
status Content status: draft, hidden, or published
template Name of template to use to generate content (without extension)
save_as Save content to this relative file path
url URL to use for this article/page

除了title外，所有元数据都不是必须的。如果没有指定，DEFAULT_DATE被设置为'fs'，pelican依赖文件的"mtime" 时间戳，catagory为文件所在目录。如果不希望如此，需要设置USE_FOLDER_AS_CATEGORY为False。

如果想自动使用文件名做title，可在设置中增加一个语句： FILENAME_METADATA = '(?P<title>.*)'

通过设置FILENAME_METADATA，你也可以通过正则表达式从文件名中抽取元数据，所有匹配的命名组都会被设置为元数据。FILENAME_METADATA的缺省设置只会从文件名中抽取日期。示例：

'(?P<date>\d{4}-\d{2}-\d{2})_(?P<slug>.*)'

页面

content\pages目录下的所有的文件都会被用来产生静态页面。

可以使用DISPLAY_PAGES_ON_MENU设置来控制是否所有这些页面都被显示在主导航菜单。缺省设置是True。如果想在此菜单中，排除任何页面，那么只需在该页面的元数据中增加status:hidden即可。

静态内容

静态文件是跟文章和页面不同，但同样被拷贝到output文件夹的文件。通过pelicanconf.py中的STATIC_PATHS设置，可以控制那些静态文件被拷贝。pelican的缺省配置包括images目录，其他需要手工加入。

同一目录下的混合内容

pelican支持把静态文件和页面文件放到同一个目录下。这种目录必须同时加入到STATIC_PATHS和PAGE_PATHS中（或STATIC_PATHS和ARTICLE_PATHS）。pelican会识别和处理页面文件，并把拷贝其他文件（就像它们在一个分开的静态文件目录中一样）

注意： 静态和页面文件放在同一个目录中，在生成的站点中，未必会在同一个位置。要想使之在一起，最简单的方式是使用{attach} 链接语法。

链接到内部内容（页面、文章）

可以在原始内容层次指定站内链接。 为了链接到内部内容（content目录下的文件），使用下述语法{filename}path/to/file。注意： filename是关键字，后面是实际的path，在所有操作系统中都要使用/。

举例：对于如下的文件结构

website/
├── content
│   ├── category/
│   │   └── article1.rst
│   ├── article2.md
│   └── pages
│       └── about.md
└── pelican.conf.py

article1.rst

The first article
#################

:date: 2012-12-01 10:02

See below intra-site link examples in reStructuredText format.

`a link relative to the current file <{filename}../article2.md>`_
`a link relative to the content root <{filename}/article2.md>`_

article2.md

Title: The second article
Date: 2012-12-01 10:02

See below intra-site link examples in Markdown format.

[a link relative to the current file]({filename}category/article1.rst)
[a link relative to the content root]({filename}/category/article1.rst)

链接到静态文件

可以使用{static}path/to/file链接到静态内容。使用此语法pelican将会自动把链接的文件拷贝到output目录，即使该文件所在目录没有包含在pelicanconf.py配置文件的STATIC_PATHS设置中。 举例：如下是项目的content目录

content
├── images
│   └── han.jpg
├── pdfs
│   └── menu.pdf
└── pages
    └── test.md

test.md内容为

![Alt Text]({static}/images/han.jpg)
[Our Menu]({static}/pdfs/menu.pdf)

站点生成时，han.jpg将被拷贝到output/images/han.jpg，menu.pdf 将被拷贝到output/pdfs/menu.pdf，并在test.md中写入适当的链接。

attaching 静态文件

语法为 {attach}path/to/file。工作方式与{static}类似。如果静态文件起初在链接源文件的子目录下，那么在output也会保持这个关系。否则，它将变成该文件的邻居。 示例：项目目录结构

content
├── blog
│   ├── icons
│   │   └── icon.png
│   ├── photo.jpg
│   └── testpost.md
└── downloads
    └── archive.zip

pelicanconf.py相关设置：

PATH = 'content'
ARTICLE_PATHS = ['blog']
ARTICLE_SAVE_AS = '{date:%Y}/{slug}.html'
ARTICLE_URL = '{date:%Y}/{slug}.html'

testpost.md相关内容

Title: Test Post
Category: test
Date: 2014-10-31

![Icon]({attach}icons/icon.png)
![Photo]({attach}photo.jpg)
[Downloadable File]({attach}/downloads/archive.zip)

生成的站点具有的output目录结构为

output
└── 2014
    ├── archive.zip
    ├── icons
    │   └── icon.png
    ├── photo.jpg
    └── test-post.html

注意： 所有使用{attach}链接的文件，都在aritcles的output目录之内，或之下。

如果一个静态文件被链接了多次，{attach}将只会在链接的第一次处理时进行拷贝。之后的{attach}链接其处理跟{statci}相同。这可以防止破坏已处理的链接。

从多个文件链接同一个文件时要小心！ 应为pelican没有定义文件处理顺序，所以每次处理的顺序可能不同。使用{attach}链接多个文件时，可能会导致output目录中相关文件的位置每次生成站点时都不一样，这可能会导致外部站点链接到这个文件的链接失效。

所以建议仅在同时满足下述条件时使用{attach}链接多个文件： 1. 所有链接都指向一个文件。 2. 源链接文件都在一个目录下。

链接到authors, catagories, index, - tags

{author}name, {category}foobar, {index}, {tag}tagname
